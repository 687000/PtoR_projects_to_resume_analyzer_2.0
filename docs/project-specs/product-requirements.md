# Product Requirements Document

**Product:** Project-to-Resume Analyzer

---

## 1. Problem Statement

Software engineers with strong project backgrounds often struggle to articulate that experience in ways that resonate with different target roles. The problem is not a lack of experience — it is translating complex, multi-party project work into accurate, role-targeted language without:

- Overclaiming responsibility
- Underselling genuine technical contributions
- Rewriting the same content from scratch for every job application

This tool solves that by structuring project knowledge once and generating reusable content on demand.

---

## 2. Target Users

**Primary user:** A software engineer who:
- Is actively job searching across roles that may require different framing
- Wants accurate, professional resume content without repeating effort per application

---

## 3. Terminology

See [GLOSSARY.md](../GLOSSARY.md) for full definitions.

| Term | Definition |
|---|---|
| **T1 responsibilities** | Technical analysis and planning: scope analysis, platform assessment, API/interface design |
| **T2 responsibilities** | Hands-on implementation: web development, detailed design, feature delivery |
| **Project context** | Background the user provides that is *not* their own work — business background, team/client requirements, PM product decisions |
| **JD** | Job Description |
| **Reusable project bullets** | Project-centric bullets generated from the project analysis workflow and stored as part of the canonical project record |
| **Tailored bullets** | JD-specific bullets generated by the matching workflow using only saved project evidence and extracted JD requirements |
| **Provenance** | Evidence references linking parsed source segments to extracted or generated outputs |

## 4. Product Goal

Convert project materials and job descriptions into interview-ready and resume-ready content through:

1. **Project analysis workflow**
2. **Job matching workflow**

---

## 5. Primary Use Cases

### Use Case A: Project Analysis

The user provides project materials together with structured context.

**Background context (not attributable to the user’s work):**

* Project background
* Business, team, or client requirements
* Prior decisions and constraints

**User contribution context:**

* T1 responsibilities
* T2 responsibilities
* Architecture involvement
* Cross-platform coordination
* Challenges, constraints, and tradeoffs
* Outcomes and impact

**System outputs:**

* Project summary
* Ownership and responsibilities
* Technical contributions and highlights
* Resume-ready bullet points (3–5)
* Technical learning arc
* Interview talking points
* Anticipated interview questions with answer frames
* STAR-format interview answer
* Supporting evidence references

### Use Case B: Job Description Analysis and Project Matching

The user provides a job description in a supported format.

**Job description inputs:**

* Plain text paste
* PDF
* Image or screenshot
* URL to a career page
* Notion page URL

**System analysis of the job description:**

* Extract tech stack signals
* Extract collaboration and leadership signals
* Extract domain and product context
* Extract seniority indicators
* Classify the role category
* Store a structured JD target with evidence references where supported

**System matching behavior:**
* Compare the new JD target against previously saved JD targets for deduplication
* If the JD is substantially similar to an existing target, prompt the user to update, save as new, or cancel
* Compare the JD target against all saved project analyses
* Calculate a fit score for each project using weighted matching dimensions
* Rank projects by overall relevance to the role

**System outputs:**
* Structured extracted requirements
* Deduplication result and similarity decision point
* Ranked project match results
* JD-specific tailored bullet points per matched project
* Export-ready resume content grouped either by project or as a flat ranked list

## 6. Module Specifications

### Module A: Project Analysis

#### 6.1 Supported Input Formats

| Format             | Method                                                     |
| ------------------ | ---------------------------------------------------------- |
| Plain text         | pass-through                                               |
| PDF                | `pdfplumber`                                               |
| Image / screenshot | OCR via Tesseract                                          |
| HTML file          | `BeautifulSoup` with content extraction                    |
| URL (website)      | `requests` + `BeautifulSoup`                               |
| Notion link        | Notion API block traversal                                 |

#### 6.2 User-Provided Context Schema

| Field                                  | Attribution                    |
| -------------------------------------- | ------------------------------ |
| Project background                     | Background only; not user work |
| Business / team / client requirements  | Background only; not user work |
| PM product decisions                   | Background only; not user work |
| T1 responsibilities                    | User contribution              |
| T2 responsibilities                    | User contribution              |
| Architecture details                   | User contribution              |
| Web / mobile / server coordination     | User contribution              |
| Challenges, constraints, and tradeoffs | User contribution              |
| Outcomes and impact                    | User contribution              |

#### 6.3 Generated Outputs

The system generates and stores the following fields for each project:

1. Professional project summary
2. Ownership and responsibility description
3. Key technical contributions and highlights
4. Resume-style bullet points (3–5 reusable, project-centric bullets)
5. Technical learning arc
6. Anticipated interview questions (4–6) with suggested answer frames
7. STAR-format interview answer
8. Short self-introduction
9. Interview talking points
10. Evidence references / provenance links, where supported

Complete schema: `OUTPUT-SCHEMA.md`

#### 6.4 Dual-Source Verification

The system must explicitly verify that both extracted source material and user-provided contribution fields are used.

Verification requirements:

* Extracted source material is used for implementation and project context
* At least one user contribution field is used as the primary signal for user-owned work
* Empty contribution fields may fall back to extracted text for technical context only, not for ownership attribution
* No responsibility is claimed unless supported by contribution fields and/or source evidence
* Provenance must remain attachable from parsed segments to downstream generated outputs

This prevents over-claiming ownership of PM decisions, team requirements, or other background context.

#### 6.5 Output Rules

* Clearly distinguish user work from PM, team, or business decisions
* Do not attribute PM product decisions to the user
* Emphasize technical analysis, platform coordination, API and interface definition, and structured web implementation
* Keep outputs professional, concise, and job-search ready
* Ensure the technical learning arc highlights insight and transfer value, not only implementation detail
* Ensure interview questions are realistic and preparation-oriented
* Store reusable bullets as project-centric, not JD-specific


### Module B: Job Description Analysis & Matching

#### 6.6 Inputs (Supported Formats)

| Format | Method |
|---|---|
| Plain text paste | Direct input |
| PDF | pdfplumber extraction |
| Image / screenshot | Tesseract OCR |
| URL (career page) | requests + BeautifulSoup |
| Notion page URL | Notion API block traversal |

#### 6.7 Extraction Logic

Extract requirement signals from JD text:

1. **Tech Stack** - languages, frameworks, databases, tools
2. **Collaboration Signals** - team types, cross-team work, leadership
3. **Domain Context** - business domains, product types, industry
4. **Seniority Signals** - experience level indicators
5. **Role Category** - role type classification

Output: Structured `extracted_requirements` object with evidence references where supported.

#### 6.8 Deduplication Check

When a new JD is uploaded:
1. Compare extracted requirements against all existing saved JD targets
2. Calculate requirement overlap percentage (keywords + theme overlap)
3. If overlap is ≥70%: prompt user to Update existing | Save as new | Cancel
4. Otherwise: proceed to matching silently

Deduplication metadata belongs to the JD-side workflow, not the project analysis workflow.

#### 6.10 Tailored Bullet Generation

For each matched project:

1. **Initial matching**: Select and rank existing project bullets by JD requirement relevance
2. **On-demand generation**: Generate new LLM-tailored bullets when user clicks "Generate Bullets"
3. Limit to 2-4 most relevant bullets per project (mix of selected and generated)

Output: `tailored_bullets` array per matched project with detailed metadata.

### Tailored Bullets Generation

**Trigger:** User clicks "Generate Bullets" button in project detail view
**Requirements:** `ANTHROPIC_API_KEY` environment variable must be configured
**Process:**
- Uses `jd-tailored-bullets-prompt` to generate 2-3 new bullets
- Addresses specific JD requirements (tech stack, domain, collaboration, seniority)
- All content traces back to existing project evidence (no fabrication)
- Generated bullets are appended with source metadata and reasoning

**Output Structure:**
```json
{
  "tailored_bullets": [
    {
      "bullet": "string (14-32 words, action verb)",
      "addresses": ["tech_stack", "domain", "collaboration", "seniority"],
      "source_references": ["resume_bullets[0]", "technical_highlights[2]"],
      "reasoning": "Why this bullet addresses JD requirements"
    }
  ],
  "generation_notes": "Limitations or trade-offs in generation"
}
```

Important boundary:

* `resume_bullets` in the project record are reusable project outputs
* `tailored_bullets` are JD-specific outputs owned by the matching workflow
* Generated bullets are preserved separately and not overwritten on re-match

#### 6.11 Resume Export

From matched results, user can:

1. Edit tailored bullets inline
2. Pin/unpin bullets for inclusion in export
3. Copy formatted resume section to clipboard
3. Export as:

   * Plain text
   * Markdown
   * PDF
4. Grouping options:

   * By project
   * Flat list sorted by fit score

## 7. Prompt and Matching Design

This product uses prompt-based generation and structured matching logic to connect saved projects with job descriptions.

### 7.1 Prompt-Based Generation

The system generates resume and interview artifacts from project records and user contributions using prompts documented under `project-specs/prompts/`.

Key design principles:

* user contributions take precedence over extracted source text
* extracted source text supplements missing detail, but does not override contribution ownership
* background context informs interpretation without being attributed as the user's work
* generated content is grounded in existing project evidence and user-provided contributions
* evidence references should remain preservable through the generation pipeline where supported

Primary output types:

* professional summaries
* reusable project bullets
* technical highlights
* STAR interview answer frames
* technical learning arcs
* anticipated interview questions
* interview talking points
* JD-tailored bullets

### 7.3 JD Extraction and Matching Logic

Job descriptions are converted into structured requirement signals that are used to compare against saved projects.

Extraction fields:

* `tech_stack`
* `collaboration_signals`
* `domain_context`
* `seniority_signals`
* `role_category`

Matching logic is designed to surface the best-fit projects by combining multiple dimensions of alignment.

Core scoring dimensions:

* tech overlap
* domain relevance
* collaboration strength
* seniority alignment

`seniority_alignment` is evaluated using JD expectations and candidate profile data stored in `data/user_profile.json`, rather than inferred only from project scope.

### 7.4 Tailored Bullet Generation

When generating JD-specific bullets, the system should:

1. base output on configured role emphasis and extracted requirements
2. use only evidence already present in the project record
3. map bullets to requirement signals where possible
4. rank bullets by relevance to the JD
5. limit output to the most relevant candidates per project

### 7.5 Verification and Safety Rules

To prevent inaccurate attribution or overclaiming:

* require extracted project source text and at least one user contribution field for project generation
* treat contributions as the primary ownership signal
* fall back to extracted source text only for additional technical detail
* do not claim PM or stakeholder decisions as the user's work
* keep prompt rules in project-specs prompt documents

## 8. Information Architecture

### A. Projects Page

Project list - a responsive card grid with:

* Project title
* Category badge
* Technical tags
* Summary excerpt
* Created date
* View / Delete actions

Search and filter:

* Real-time text search across title, summary, tags
* Category dropdown filter
* Tag dropdown filter

Project upload area - a collapsible panel

* Input tabs: Text | File | URL | Notion
* Context form (Background / Contributions)

* Source text preview

* Tabs: Source & Context | Summary | Highlights | Resume Bullets | Interview
